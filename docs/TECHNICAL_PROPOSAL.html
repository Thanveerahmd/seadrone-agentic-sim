1. Executive Summary

This document defines the complete technical architecture for GPS-denied multi-drone maritime target triangulation. Two autonomous sea drones (Agent A, Agent B) collaboratively locate, track, and triangulate a target vessel using only onboard sensors — camera (YOLOv8-nano + SAMURAI tracker), IMU, compass — and inter-agent communication via a ground station orchestrator or direct mesh link.

Core Innovation: A single drone can determine the bearing (direction) to a target but NOT the range (distance). Two drones observing from different positions produce two bearing lines whose intersection resolves the target's position — without any GPS on the target.

Key validated capabilities:

  • Dead reckoning self-positioning: 3.4% drift over 800m (field-validated)
  • Camera-based target bearing: directionally accurate (YOLO/SAMURAI validated)
  • Target loss recovery: <30 seconds via coordinated pincer search
  • System latency: <250ms end-to-end (detection to drone response)
  • Communication: <64 bytes/sec per drone (works on any radio link)

3.3 Dead Reckoning Feasibility (Field Validated)

TestMethodResult
Self-Position (DR vs GPS)Compass heading + ground speed integration3.4% drift over 800m in 5 minutes
Target BearingYOLO/SAMURAI bbox centroid + drone headingDirectionally accurate
Target RangeBbox size → range estimateApproximate (±30-50%) — needs triangulation
Key finding: A single drone can determine target direction but NOT target distance. This is the fundamental reason two-drone triangulation is required — two bearings from different positions geometrically resolve the range.

DR Error Model for Triangulation

DR position error after traveling distance D:
  σ_position = 0.034 × D

Example: drone traveled 200m since last known fix
  σ_position = 6.8m
  At 50m target range → ~7.7° bearing line origin error
  At 100m target range → ~3.9° bearing line origin error

Mitigation: weight each drone's bearing by inverse of DR age

4. The GPS-Denied Problem

4.1 Known vs Unknown

Known (per drone)Unknown
Own position (dead reckoning from launch)Target position (world coordinates)
Own heading (compass + IMU yaw)Target heading
Own speed (ground speed sensor)Target speed
Target bearing in camera (bbox centroid)Target range/distance
Target pixel velocity (px/s)Target world velocity
Detection confidenceTarget identity confirmation

4.2 Why Triangulation Solves the Range Ambiguity

Single Drone:
  ✓ Own position (DR, 3.4% drift)
  ✓ Target BEARING (camera) — accurate direction
  ✗ Target RANGE — unknown

Two Drones:
  ✓ Both own positions (each via DR)
  ✓ Two independent BEARINGS from different positions
  ✓ Bearing intersection → TARGET POSITION resolved
  ✓ RANGE determined geometrically — no calibration needed

5. Coordinate System — GPS-Free Local Frame

All positions expressed in meters East/North from the ground station (launch point). No latitude/longitude anywhere in the protocol.

Origin:  Ground Station position = (0, 0)
X-axis:  East (positive = right)
Y-axis:  North (positive = up)

Drone A launch: (-3, 0) meters from GS
Drone B launch: (+3, 0) meters from GS

Position update (each timestep dt):
  x += speed × cos(heading) × dt
  y += speed × sin(heading) × dt

Position quality tag (sent with every message):
  dr_distance_m:     total distance traveled on DR
  estimated_error_m: dr_distance × 0.034
  age_sec:           time since last absolute calibration

6. Bearing Estimation from Camera Data

6.1 Camera-to-Bearing Conversion

Target bbox center: (cx, cy) in pixels
Camera FOV horizontal: FOV_H = 60°
Frame width: W = 960 pixels

Pixel offset from center: dx = cx - (W / 2)
Bearing offset: θ_offset = (dx / (W/2)) × (FOV_H / 2)
Absolute bearing: θ_target = drone_heading + θ_offset

Example from real data:
  bbox center x = 626, heading = 152°
  dx = 626 - 480 = +146px (target right of center)
  θ_offset = (146/480) × 30° = +9.1°
  θ_target = 152° + 9.1° = 161.1°

6.2 Bearing Confidence & Wave Compensation

corrected_bearing = raw_bearing - roll_component × sin(pitch)
bearing_confidence = detection_confidence × (1 - |roll| / 30°)

From real data: roll up to 5.6°, pitch up to 8.6°
→ Systematic bearing error correctable with IMU data

6.3 Bearing Rate for Target Motion Prediction

bearing_rate = (velocity_px_per_sec.x / (W/2)) × (FOV_H / 2)   deg/sec

Real data: velocity_px_per_sec.x ranges -274 to +90 px/s
→ bearing_rate ranges roughly -17 to +5.6 deg/sec

This tells us:
  Positive rate → target moving right in camera → moving left relative to drone heading
  Negative rate → target moving left in camera → moving right relative to drone heading
  High |rate| → target crossing fast → LEAD INTERCEPT required

8. Triangulation Algorithm

8.1 Bearing-Only Intersection

Agent A at (Ax, Ay) with bearing θa
Agent B at (Bx, By) with bearing θb

Bearing line A: (x,y) = (Ax,Ay) + t × (cos θa, sin θa)
Bearing line B: (x,y) = (Bx,By) + s × (cos θb, sin θb)

Intersection parameter:
  t = ((Bx-Ax)×sin θb - (By-Ay)×cos θb) / (cos θa×sin θb - sin θa×cos θb)

Target position:
  Tx = Ax + t × cos θa
  Ty = Ay + t × sin θa

9. Agentic Path Planning

9.1 Predictive Bearing-Rate Intercept (PBRI)

Key principle: Don't fly to where the target IS — fly to where it WILL BE. Use bearing rate history to predict target motion and compute an optimal intercept point that maximizes triangulation geometry. 
PBRI Algorithm:

Step 1: ESTIMATE TARGET MOTION
  Input: N bearing reports from Agent A over time window T
  Compute: bearing rate ω = dθ/dt (linear regression)
  Classify: |ω| < 0.5°/s → STATIONARY
            |ω| 0.5-3°/s → SLOW
            |ω| 3-8°/s   → MEDIUM
            |ω| > 8°/s   → FAST / TURNING

Step 2: COMPUTE PREDICTED TARGET ZONE
  Target bearing at T_future: θ_future = θ_now + ω × ΔT
  Range estimate: R = K / √(bbox_area)
  Predicted position: P = P_A + R × (cos θ_future, sin θ_future)
  Uncertainty: ±5° + 2°/sec × time_since_detection

Step 3: SCORE INTERCEPT CANDIDATES
  For each candidate arrival time T:
    score = w1 × triangulation_quality(angular_sep)
          + w2 × arrival_margin(time_slack)
          + w3 × fov_coverage(zone_size vs FOV)
          - w4 × energy_cost(distance / battery)

Step 4: GENERATE CURVED PATH
  Dubins path from B's current position to intercept point
  Arrive facing the predicted target zone (FOV pre-aimed)

9.2 Target Loss Recovery — Expanding Pincer Search

Phase 1: IMMEDIATE (0-5 sec)
  A: Turn toward last bearing rate direction
  A: Broadcast TARGET_LOST with escape prediction
  B: Receive and compute predicted escape zone

Phase 2: COORDINATED SEARCH (5-30 sec)
  Escape zone = wedge from last known position
  A: Search LEFT half of escape wedge
  B: Search RIGHT half of escape wedge
  → Entire probable zone covered, no overlap

Phase 3: RE-ACQUISITION
  Whichever drone finds target → BEARING_REPORT
  Other drone → intercept for triangulation

9.3 Energy-Optimal Speed Profile

Trapezoidal profile:
  Phase 1 (20%): Accelerate — smooth heading transition
  Phase 2 (60%): Cruise at optimal speed
  Phase 3 (20%): Decelerate to tracking speed (stable camera)

Speed selection:
  Short (<50m):   2.5 m/s (accuracy priority)
  Medium (50-100m): 3.0 m/s (balanced)
  Long (>100m):    3.5 m/s (speed priority)
  Critical:        3.6 m/s max
